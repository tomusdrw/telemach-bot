# Verbatim subject for short messages + AI body expansion

**Date:** 2026-07-08
**Status:** Approved, pending implementation plan

## Summary

Two coupled changes to how an approved user's message becomes an email:

1. **Short message → subject 1-1.** If the message text is short (single line, ≤ 80 chars), it becomes the email subject verbatim — no AI paraphrase. Longer messages keep today's behaviour (AI-generated subject).
2. **AI-cleaned/expanded body (always).** For every message that has text, an AI pass produces a tidied version — fix typos, expand terse phrasing into a simple full sentence — which is appended **below** the original in the body. The original text is never replaced (additive). Links are left exactly as-is; the AI is explicitly told not to interpret or modify URLs.

Both AI passes are best-effort: on any failure they degrade gracefully (verbatim/AI subject fallback; expansion omitted). Neither may delay or fail an otherwise valid email — this extends the spec's existing "subject generation is the exception" rule to the expansion pass too.

## Motivation

Today the AI compresses the message into a subject and the raw message goes in the body. For short notes the AI paraphrase loses the user's exact wording. The user wants their exact short note as the subject, and a lightly-cleaned/expanded rendition of the message in the body.

## Goals

- Short, single-line messages appear verbatim as the email subject.
- Every text-bearing message gets an AI-cleaned/expanded rendition appended to the body.
- Original message text is always preserved in the body (additive).
- URLs are passed through untouched — no fetching, no summarising, no rewriting.
- No new failure mode can block or delay a valid email.

## Non-goals (explicitly out of scope)

- **Link fetching / page summarisation.** Considered and dropped. It is a separate subsystem (arbitrary-URL fetch, HTML cleanup, SSRF surface, latency, dead/paywalled/PDF/SPA handling) and would be its own spec if ever wanted. The expansion prompt actively forbids touching links.
- Replacing the original message text with the AI version (we chose additive).
- Per-user toggles for either behaviour.
- Deduplicating near-identical (but not exactly-equal) long expansions.

## Behaviour

### Subject

`body` is the existing derived text: `transcribedBody ?? captions.join('\n\n')` (unchanged). Voice transcripts are treated uniformly — same rules apply to the transcript text.

- **Short** is decided by `isShortSubject(text)` (pure): after `trim()`, the text is non-empty, contains **no newline**, and is **≤ 80 chars**.
- If short → subject = `verbatimSubject(text)` (pure): `trim()` + hard cap at 80 chars. No trailing-punctuation stripping, no quote stripping — it is the user's literal text. **The AI subject call is skipped.**
- If not short (including the empty-body / attachments-only case, which keeps using `attachmentCountLabel(...)` / `(no text)`) → subject is generated by the existing AI path with the existing `fallbackSubject` fallback.

80 is chosen as the threshold because it is the existing subject length cap, so a "short" message is exactly one that fits a subject line unmodified.

### Body expansion

New pure module `src/bot/body-expansion-prompt.ts`:

- `buildExpansionPrompt(body)` — instructs the model to:
  - Fix typos and expand terse phrasing into a simple, full sentence.
  - Reply in the **same language** as the message (do not translate).
  - **Leave all URLs/links exactly as-is** — do not interpret, follow, summarise, or reword them.
  - Not invent facts, names, or dates that are not present in the message.
  - Reply with the rewritten text only, no preamble.
- `sanitizeExpansion(raw)` — trims; returns `''` for empty. (Kept minimal; body is free-form text, unlike the subject.)

New service `src/services/body-expansion.ts`, mirroring `services/subject.ts`:

- `expand(body): Promise<string | null>` — single OpenRouter chat-completion attempt. Returns `null` on non-2xx, unexpected shape, or thrown error. **Never throws.**
- Options: `{ apiKey, model, fetchImpl? }`.

### Body composition (additive)

In `forward.ts`, the expansion call is added to the existing `Promise.all` (alongside subject + event extraction) so latency does not accumulate. It runs whenever `body` is non-empty, wrapped in `tryOrNull` for defence in depth.

Assembled body order: **original → separator → expansion → (existing event note)**.

```
kup mleko

———
✍️ Kup mleko.

📅 Event attached: ...
```

- Separator/label: a line of em-dashes (`———`) followed by `✍️` prefix on the expansion. Approved; trivial to change later.
- **Skip appending** when the expansion, normalised (lowercase, punctuation and whitespace stripped), is **exactly equal** to the normalised original — avoids a pure duplicate for the common short-message case.
- Known limitation (accepted): for a long, already-well-written message, "always expand" yields a near-duplicate block that exact-equality normalisation will not catch. The user accepted this cost when choosing "always expand".
- The event note continues to be appended after the (possibly expansion-augmented) body, exactly as today.

### Error model

- `expand` returns `string | null` and never throws — same contract as `subject.generateSubject`. A `null` (or an exact-duplicate expansion) means "no expansion section is added"; the email sends with the original body.
- This is a deliberate extension of the spec's "subject generation is the exception" clause. Document it in `CLAUDE.md` / the original design doc's error-model section.

## New / changed modules

| File | Change |
|------|--------|
| `src/bot/subject-prompt.ts` | add `isShortSubject(text)`, `verbatimSubject(text)` (pure) |
| `src/bot/body-expansion-prompt.ts` | **new** — `buildExpansionPrompt`, `sanitizeExpansion` (pure) |
| `src/services/body-expansion.ts` | **new** — `makeBodyExpansionClient` / `expand` (mirrors `subject.ts`) |

| `src/bot/index.ts` | construct expansion client, pass into `makeForwardHandler` |

`ForwardDeps` gains `expansion: BodyExpansionClient`.

Chosen approach: **mirror the existing `subject` / `event-extraction` pattern** rather than first refactoring a shared `openrouterChat()` helper. The shared-helper refactor touches two working, unrelated files and is out of scope; it could be a separate chore later.

## Testing (TDD)

- `subject-prompt`: `isShortSubject` — 80-char boundary, newline → false, empty/whitespace → false; `verbatimSubject` — trim, cap 80, punctuation preserved.
- `body-expansion-prompt`: prompt contains the links-as-is rule and same-language rule; `sanitizeExpansion` trims / empties.
- `body-expansion` (mocked fetch): returns content; `null` on non-2xx; `null` on thrown error.
- `forward`:
  - short text → subject equals verbatim message **and** AI subject client is not called;
  - long text → subject uses AI path (unchanged);
  - expansion appended below original with separator;
  - exact-duplicate expansion is skipped;
  - `null` expansion → body is original only;
  - expansion failure/throw does not fail or delay the email;
  - short voice transcript → verbatim subject + expansion (uniform handling).

All tests mock at the SDK/fetch boundary, consistent with the existing suite (no real providers in CI).
