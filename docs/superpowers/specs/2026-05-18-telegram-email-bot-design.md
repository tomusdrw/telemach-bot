# Telegram → Email Bot — Design

**Status:** Draft
**Date:** 2026-05-18
**Owner:** @tomusdrw

## Purpose

A personal Telegram bot that forwards anything you send it to your email inbox. Text becomes the email body; attachments are forwarded verbatim; voice messages are transcribed by Whisper and sent as text; the email subject is generated by an LLM via OpenRouter. Used by a small allowlist of approved users, each routing to their own destination email.

The bot never replies in chat for the forwarding flow — it only uses Telegram reactions on the original message to indicate progress and completion. Replies are only used during onboarding/admin flows.

## Goals

- One Telegram message in → one email out, with minimal latency.
- Multi-user with admin approval; each user has their own destination email.
- Reaction-based feedback so the chat stays clean.
- Runs as a single Docker container on a small VPS with one mounted data volume.
- Pluggable external providers (Whisper, OpenRouter model, Resend) via env vars.

## Non-goals (v1)

- Files larger than 20 MB (Bot API limit; requires self-hosted Telegram Bot API server).
- Per-user model/provider preferences.
- Editing previously-sent emails when a Telegram message is edited.
- Email threading; each message becomes a standalone email.
- Multi-recipient routing for one user.
- Inbound replies (replying to the bot's email does *not* round-trip to Telegram).
- Web UI / admin panel — admin actions happen in Telegram only.
- Encryption at rest beyond what the host's disk provides.
- Cost tracking / usage limits per user.
- Internationalization — bot text is English.

## High-level architecture

Single Node.js/TypeScript process. Long-polling against the Telegram Bot API (no public URL, no inbound webhook). All persistent state lives in one SQLite file under `/data` inside the container, mounted from the host. External providers (OpenAI Whisper, OpenRouter, Resend) are accessed via thin wrappers.

```
┌─────────────────────────────────────────────────────────┐
│                  bot process (single)                   │
│                                                         │
│  grammy ──► auth middleware ──► router                  │
│                                       │                 │
│                ┌──────────────────────┼──────────────┐  │
│                ▼                      ▼              ▼  │
│           onboarding             admin               │  │
│           (/start,               callbacks           │  │
│            /register)            (approve/reject)    │  │
│                                                      │  │
│                            forwarded-message handler ◄  │
│                                       │                 │
│                                       ▼                 │
│                              media-group buffer         │
│                              (in-memory, keyed by       │
│                               media_group_id)           │
│                                       │                 │
│                       ┌───────────────┼───────────────┐ │
│                       ▼               ▼               ▼ │
│                   Whisper         OpenRouter        Resend
│                   (voice)         (subject)         (email)
│                                                         │
│                  SQLite ◄── user store, audit log       │
└─────────────────────────────────────────────────────────┘
```

### Why long-polling, not webhooks

Telegram offers two delivery modes only: long-polling (`getUpdates`) and webhooks. For a single-instance personal bot on a VPS, long-polling is the simpler choice: no public URL, no TLS termination, no webhook secret. Telegram holds the connection open up to ~50 seconds, so an idle bot makes roughly one HTTP request per minute. grammy's runner handles retries, backoff, and graceful shutdown. If usage ever scales past a single instance, switching to webhooks is a one-line change.

## Tech stack

- **Runtime:** Node.js 22 (LTS at time of writing)
- **Language:** TypeScript (strict)
- **Bot framework:** grammy
- **Database:** SQLite via `better-sqlite3`
- **HTTP/SDK clients:** `openai` (for Whisper), `resend`, plain `fetch` for OpenRouter (its API is OpenAI-compatible but a thin wrapper avoids pulling extra deps)
- **Logging:** `pino`, JSON to stdout
- **Validation:** `zod` (env config, OpenRouter response shape)
- **Tests:** `vitest`
- **Packaging:** Docker, two-stage build, runs as non-root

```
│   ├── admin.ts             # admin DM + approve/reject callbacks
│   ├── forward.ts           # main message handler

│   └── media-group.ts       # in-memory buffer + flush timer
├── services/
```

Each external-service wrapper exposes a single function and is mockable in tests. The `bot/*` files contain handlers only; business logic that's worth unit-testing (subject prompt, media-group state machine, auth predicates, email composer) is broken out into pure modules.

## Data model (SQLite)

```sql
CREATE TABLE IF NOT EXISTS users (
  telegram_id   INTEGER PRIMARY KEY,
  username      TEXT,
  first_name    TEXT,
  email         TEXT,
  status        TEXT NOT NULL
                 CHECK (status IN
                   ('PENDING_EMAIL','PENDING_APPROVAL','APPROVED','REJECTED')),
  is_admin      INTEGER NOT NULL DEFAULT 0,
  created_at    INTEGER NOT NULL,
  updated_at    INTEGER NOT NULL
);

CREATE TABLE IF NOT EXISTS audit_log (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  telegram_id     INTEGER NOT NULL,
  chat_message_id INTEGER,
  event           TEXT NOT NULL,   -- 'received','transcribed','emailed','error'
  details         TEXT,            -- freeform / JSON
  created_at      INTEGER NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_audit_user_time
  ON audit_log (telegram_id, created_at DESC);
```

Schema is applied at startup with idempotent `CREATE TABLE IF NOT EXISTS` statements. No migration framework — overkill for two tables.

### Admin bootstrap

On startup, if no row exists for `ADMIN_TELEGRAM_USER_ID`, insert one with `status='APPROVED'`, `is_admin=1`, `email=ADMIN_EMAIL`. Idempotent across restarts.

### User status machine

```
(no row) ── /start ──► PENDING_EMAIL
PENDING_EMAIL ── /register <email> ──► PENDING_APPROVAL
PENDING_APPROVAL ── admin approves ──► APPROVED
PENDING_APPROVAL ── admin rejects ──► REJECTED
```

`APPROVED` is terminal in v1 (no revocation flow); admin can edit the row directly in SQLite if needed.

## Data flows

### Onboarding (new user)

```
new user → /start
  └─► db: insert row with status=PENDING_EMAIL
  └─► bot replies: "Hi. Reply with /register your@email.com to get started."

user → /register me@example.com
  └─► validate email format
  └─► db: status=PENDING_APPROVAL, email=me@example.com
  └─► bot replies to user: "Got it. Waiting for admin approval."
  └─► bot DMs admin with inline keyboard:
        "New user request:
         @username (id: 12345)
         email: me@example.com"
        [Approve] [Reject]

admin → taps Approve
  └─► db: status=APPROVED
  └─► bot edits admin's DM to "Approved @username ✓"
  └─► bot DMs the user once: "You're approved. Send away."

admin → taps Reject
  └─► db: status=REJECTED
  └─► bot edits admin's DM to "Rejected @username ✗"
  └─► bot DMs the user once: "Your request was declined."
```

Onboarding is the **only** flow where the bot sends plain text replies. Anything from an `APPROVED` user is acknowledged solely via reactions.

### Behavior for non-approved users

| Sender status | Non-command message | `/start` | `/register <email>` |
|---|---|---|---|
| no row | silently ignored | creates row, goes to PENDING_EMAIL | creates row, goes to PENDING_APPROVAL |
| PENDING_EMAIL | reply once: "Please run /register your@email.com first." | no-op | go to PENDING_APPROVAL |
| PENDING_APPROVAL | silently ignored | no-op | replace email, stay PENDING_APPROVAL (re-DM admin) |
| REJECTED | silently ignored | silently ignored | silently ignored |
| APPROVED | normal forward flow | reply: "You're already set up." | reply: "You're already set up. Email cannot be changed from here." |

### Single forwarded message (text or single attachment)

```
1. setMessageReaction(👀)
2. if media (photo/document/voice/etc.): download via getFile (guard: ≤ 20 MB)
3. setMessageReaction(✍)
4. if voice: transcript = whisper.transcribe(file)
5. body = (text || caption || transcript || "(no text)")
6. subject = openrouter.generateSubject(body)
7. resend.send({ from: RESEND_FROM_EMAIL, to: user.email,
                  subject, html/text: body, attachments: [...] })
8. setMessageReaction(👍)
```

On any failure: `setMessageReaction(💩)` + log line.

### Voice message specifics

- File is downloaded via `getFile`, decrypted/streamed to OpenAI Whisper.
- Original audio is **not** attached to the email; only the transcript is sent.
- If Whisper returns an empty string, we treat it as `FatalError` ("could not transcribe") and react 💩.

### Media group (multiple files in one upload)

```
update #1 arrives, media_group_id="abc"
  └─► setMessageReaction(👀) on update #1
  └─► buffer["abc"] = { messages: [u1], timer: 2s }

update #2 arrives, media_group_id="abc"
  └─► setMessageReaction(👀) on update #2
  └─► buffer["abc"].messages.push(u2); reset timer to 2s

[2 seconds of silence]
  └─► flush buffer["abc"]
  └─► setMessageReaction(✍) on each
  └─► download all files
  └─► caption_text = concatenated non-empty captions
  └─► subject = openrouter.generateSubject(caption_text + " (N attachments)")
  └─► resend.send({ to, subject, attachments: [...N files], html: captions })
  └─► setMessageReaction(👍) on each
```

The buffer is in-memory only. If the process restarts mid-buffer those messages are lost and the user resends. Acceptable trade-off for personal-bot scale.

Bundling key: `media_group_id` only. Two unrelated rapid messages from the same user are *not* bundled.

### Non-photo attachments

All attachment types (document, video, voice-as-attachment, animation, sticker) are downloaded via `getFile` and attached to the email verbatim with their original filename/MIME type. Stickers and animations are forwarded as the underlying file (`.webp`, `.tgs`, `.mp4`).

## Reaction state machine

| Reaction | Meaning |
|----------|---------|
| 👀 | received / processing started (set immediately) |
| ✍  | working (transcribing voice or calling LLM) |
| 👍 | email sent successfully |
| 💩 | error (details in server logs) |

`bot/reactions.ts` exposes `markReceived(ctx)`, `markWorking(ctx)`, `markDone(ctx)`, `markFailed(ctx)`. Each sets exactly one emoji via `setMessageReaction` (replacing the previous). If the reaction call itself fails (e.g., message deleted), the error is swallowed and logged.

## Subject generation (OpenRouter)

```
prompt: "Generate a concise, descriptive email subject (max 80 chars, no
         quotes, no trailing punctuation) for the following message body.
         Reply with the subject only, no preamble.

         BODY:
         <body>"
```

- Model defaulted via `OPENROUTER_MODEL=google/gemini-flash-1.5` (cheap + fast); configurable.
- Response is trimmed, truncated to 80 chars, leading/trailing quotes stripped.
- **Single attempt, no retry.** If the call fails or returns empty, use the fallback subject `"Telegram message from @username"`. Subject generation is non-critical and must not delay email delivery; this is the one step that does *not* go through the TransientError retry loop.

## Email composition (Resend)

- `from`: `RESEND_FROM_EMAIL` (must be a verified Resend domain). Resend forbids spoofing the From address.
- `to`: user's registered destination email.
- `subject`: from OpenRouter (with fallback above), prefixed with `[TG]` (prefix is a fixed design choice — easy to change in one place).
- Body: HTML for text/captions (escaped), plain-text part also included. Sender attribution line at the top: `Sent by @username (Telegram) at <ISO timestamp>`.
- `attachments`: array of `{ filename, content: Buffer }`.


## Docker

Two-stage build for a small image; runs as non-root; one volume.

```dockerfile
# build stage
FROM node:22-bookworm-slim AS build
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci
COPY tsconfig.json ./
COPY src ./src
RUN npm run build && npm prune --omit=dev

# runtime stage
FROM node:22-bookworm-slim
WORKDIR /app
ENV NODE_ENV=production
COPY --from=build /app/node_modules ./node_modules
COPY --from=build /app/dist ./dist
COPY package.json ./
USER node
VOLUME ["/data"]
CMD ["node", "dist/index.js"]
```

```yaml
# docker-compose.yml
services:
  telemach-bot:
    image: telemach-bot:latest
    restart: unless-stopped
    env_file: .env
    volumes:
      - ./data:/data
```

`/data` is the only persistent surface. The SQLite file lives there and survives container rebuilds.

## Error handling

Two error categories, mapped explicitly:

- **TransientError** — network blip, 5xx from any provider, Telegram rate-limit (429). Retried with exponential backoff (3 attempts: 0.5s, 2s, 8s). If still failing, the message becomes a fatal failure.
- **FatalError** — 4xx from any provider, file too large, Whisper returned empty, invalid email, etc. No retry.

Either way, the final state for a failed message is `setMessageReaction(💩)` + a structured log line with error class, message id, and sanitized provider response.

The bot does **not** crash on per-message errors; every handler is wrapped in a top-level try/catch. Genuine programmer errors (e.g., DB lost) crash the process; Docker's `restart: unless-stopped` brings it back.

## Logging & observability

- Structured JSON logs to stdout via pino.
- Every inbound update logs `{event:'received', telegram_id, message_id, kinds:['photo','caption']}`.
- Every email send logs `{event:'emailed', telegram_id, message_id, resend_id}`.
- Errors log `{event:'error', error_class, provider, sanitized_response}`.
- `audit_log` table mirrors the same events for at-a-glance debugging without scrolling stdout.

## Testing strategy

1. **Pure unit tests** (vitest):
   - Subject-prompt builder.
   - Media-group buffer state machine (arrive → arrive → flush).
   - Auth predicates (status × command → allow/deny).
   - Email composer (message + attachments → Resend payload).
2. **Service wrappers** tested by mocking SDKs at the boundary. One happy-path + one error-path per service.
3. **One end-to-end-ish test** for the forward handler: construct a fake grammy `Context`, stub the three external services, run the handler, assert reactions in the right order and the expected Resend payload. No real network.

Out of test scope:
- Live Telegram bot interactions — manual smoke test after deploy.
- Real OpenAI/OpenRouter/Resend network calls.

## Open questions / future work

- **Multi-instance / high availability.** Long-polling allows only one consumer. If needed later: switch to webhooks behind a load balancer with sticky message routing.
- **Persistent media-group buffer.** Could move buffer to SQLite if in-memory loss becomes a real problem.
- **Email-reply round-trip.** Resend inbound + a simple matching scheme could let you reply to the email and have it appear in Telegram. Out of scope for v1.
- **Status revocation.** Admin currently can't un-approve a user via the bot; would need a `/users` command or similar.
